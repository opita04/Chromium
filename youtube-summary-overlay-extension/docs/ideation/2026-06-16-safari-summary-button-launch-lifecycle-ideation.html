Ideation: Safari Summary Button Launch Lifecycle

The focused problem: after Safari support was added, clicking the YouTube summary button now opens the overlay immediately. The prior and documented behavior is that the small button shows progress while the extension loads/extracts/summarizes, then the modal appears only after a cached or newly generated summary is ready.

Date: 2026-06-16 Topic: safari-summary-button-launch-lifecycle Focus: Safari summary button launch timing Mode: repo-grounded

Grounding Context

  • The README explicitly promises that progress stays on the button and the large modal opens only after the summary is ready.
  • The current YouTube content script violates that invariant in showOverlay() by setting buildOverlay().hidden = false before cache lookup, existing-summary lookup, or summary generation.
  • The current runSummary() function has an openWhenReady parameter, but its default and call site invert the desired behavior: showOverlay() passes openWhenReady: false, and runSummary() opens the modal immediately when that value is false.
  • Safari support introduced a browser-direct OpenRouter path in background/service-worker.js when direct credentials are available, while Chrome/Brave can still use the native host path. That means readiness, save ability, and cache lookup now vary by backend capability.
  • Prior repo memory says visible progress stalls should be treated as runtime/cache/native-host behavior, not syntax-only issues. In this repo, 35% means transcript is ready and the extension is waiting on SUMMARIZE_AND_SAVE; cached per-video overlay state can also preserve older behavior.
  • External context: Apple documents Safari Web Extensions as sharing common extension formats and APIs with Chrome/Firefox-style extensions, while Safari native-app messaging is a separate Safari app-extension path. That supports treating Safari as compatible at the content-script/UI layer but different at the local native integration boundary.

Ranked Ideas

1Restore Deferred Modal Opening

Axis: Launch timing contractConfidence: 92%Complexity: Low

DescriptionMake the YouTube button click start work in the background and keep the modal hidden until a cached, existing, or newly generated summary has markdown. The small circular button remains the visible progress surface at 5/10/35/50/65/78/90/100. On success, the modal opens; on failure, open it only if an error needs readable detail.

Basisdirect: README.md says the modal opens only after the summary is ready; content/youtube-transcript.js currently opens it at the start of showOverlay() and again from runSummary() when openWhenReady is false.

RationaleThis is the smallest idea that directly restores the expected behavior without forcing a larger Safari architecture decision. It also matches the user's report exactly.

DownsidesError reporting needs care: hiding the modal during work means failures need either a concise button/error state or a deliberate modal-open-on-error path.

2Make Summary Lifecycle a Small State Machine

Axis: Progress and error visibilityConfidence: 84%Complexity: Medium

DescriptionReplace boolean combinations like busy, force, and openWhenReady with explicit phases: idle, checking-cache, checking-existing, extracting, summarizing, ready, and error. Each phase declares whether the overlay is allowed to be visible and what the action button should show.

Basisdirect: showOverlay() and runSummary() currently share responsibility for cache lookup, existing-summary lookup, modal visibility, button state, progress, and generation. The bug comes from the visibility responsibility leaking into the wrong phase.

RationaleThe current regression is a symptom of state being implicit. A small lifecycle model would make Safari and Chrome differences easier to handle without reintroducing a half-loaded modal later.

DownsidesMore code than the direct fix. It should be done only if the team expects more Safari/native-host branching soon.

3Add a Capability-Aware Safari/Chrome Backend Adapter

Axis: Safari versus Chrome capability boundaryConfidence: 80%Complexity: Medium

DescriptionIntroduce a single backend capability result from the service worker: browser-direct, native-host, or unavailable. The content script can then decide what to check, what to skip, and what copy to show without using Safari-specific comments as control flow.

Basisexternal: Apple's Safari Web Extensions docs position Safari extensions as using common web-extension formats and APIs, while Apple's native app messaging docs describe a Safari app/native messaging route. direct: this repo now has both direct OpenRouter and native-host branches in background/service-worker.js.

RationaleSafari compatibility should not mean content scripts guessing which local capabilities exist. A capability adapter keeps the launch timing fix clean and gives later Safari work a stable place to live.

DownsidesRequires touching service-worker and content-script message contracts. It is not necessary to fix the immediate premature-open bug.

4Separate Cached-Ready Opens from Generate Opens

Axis: Cache and saved-summary behaviorConfidence: 82%Complexity: Low

DescriptionSplit the current click behavior into two named paths: openReadySummary() for cached/existing markdown and generateThenOpenSummary() for fresh work. Cache hits can open instantly; cache misses never open an empty modal. This keeps the good fast path without compromising the fresh-generation path.

Basisdirect: readCachedSummary(), readExistingSavedSummary(), and runSummary() already exist, but the current wrapper opens the overlay before it knows which path applies. Memory also notes cached overlay state is per-video under youtubeSummaryOverlay:<videoId>.

RationaleThe user's old behavior probably felt instant on already-summarized videos and quiet on new videos. Naming those two paths prevents the Safari shortcut from flattening them into one "open first" flow.

DownsidesStill leaves more implicit state than the state-machine idea, so future branches may need another pass.

5Add No-Premature-Overlay Regression Tests

Axis: Regression verificationConfidence: 78%Complexity: Medium

DescriptionAdd a content-script VM or lightweight DOM test proving that, on a cache miss, clicking the button updates progress but does not unhide the overlay until after SUMMARIZE_AND_SAVE returns markdown. Add paired cases for cache hit, existing summary hit, direct-mode miss, native-host miss, and failure.

Basisdirect: the repo already has tests/youtube-transcript-content.test.js using Node's vm to exercise the content script. The current bug is fully expressible as DOM state plus mocked extension messages.

RationaleSafari was added under pressure, and lifecycle regressions are easy to reintroduce. A focused test would make the README promise executable.

DownsidesThe current content script is not deeply test-harnessed, so mocking DOM and Chrome APIs may be a little fiddly.

6Add Safari-Mode Runtime Diagnostics

Axis: Safari versus Chrome capability boundaryConfidence: 70%Complexity: Low

DescriptionAdd concise debug logging around click, cache hit/miss, backend mode, progress phase, response result, and overlay visibility transitions. The log should make it obvious whether Safari used browser-direct mode, skipped Obsidian lookup, opened from cache, or opened after a new summary.

Basisdirect: prior issues in this repo were diagnosed through runtime logs and cache state; the current code has a new browser-direct branch that cannot write to Obsidian and intentionally skips native lookup.

RationaleThis gives fast proof after the fix lands. It is especially useful because Safari extension debugging can make background/content message timing harder to trust from source inspection alone.

DownsidesDiagnostics do not fix the behavior by themselves. Logging should be easy to disable or quiet once the Safari path stabilizes.
